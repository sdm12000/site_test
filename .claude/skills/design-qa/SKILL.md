# 디자인 QA 스킬

구현된 웹사이트가 "세계적 디자인 에이전시의 작품" 수준인지 냉정하게 검증한다. 핵심 질문: **"이 사이트를 Awwwards에 제출할 수 있는가?"**

## 검증 워크플로우

### Step 1: 빌드 검증

```bash
npm run build
```

빌드가 실패하면 코드 검증은 무의미하다. 빌드 에러를 먼저 해결해야 한다.

### Step 2: 디자인 시스템 충실도 검증

디자인 시스템 문서(`_workspace/02_design_system.md`)와 실제 코드를 **동시에 열어** 교차 비교한다.

| 검증 항목 | 확인 방법 |
|----------|----------|
| 컬러 값 | 디자인 시스템의 HSL/oklch 값 ↔ globals.css의 @theme 값 |
| 폰트 | 디자인 시스템의 폰트명 ↔ lib/fonts.ts의 import |
| 타입 스케일 | 디자인 시스템의 clamp() 값 ↔ CSS의 font-size |
| 스페이싱 | 디자인 시스템의 spacing scale ↔ 컴포넌트의 p/m/gap |
| 이징 커브 | 디자인 시스템의 cubic-bezier ↔ 애니메이션 코드의 ease |
| 컴포넌트 스타일 | 디자인 시스템의 버튼/카드 정의 ↔ ui/ 컴포넌트 구현 |

### Step 3: AI 탈취 검증 (가장 중요)

**"이 사이트가 AI가 만든 것처럼 보이는가?"를 10점 만점으로 채점한다. 7점 이상이 목표.**

#### AI 패턴 감지 체크리스트

각 항목을 PASS / FIX / CRITICAL로 판정한다:

**레이아웃:**
- [ ] 모든 섹션이 중앙 정렬 + 동일 패딩이면 → FIX (섹션별 변주 필요)
- [ ] 3컬럼 동일 카드 그리드가 있으면 → FIX (Bento/비정형 그리드로 대체)
- [ ] 모든 콘텐츠가 단일 컨테이너 안에 갇혀 있으면 → FIX (풀블리드 섹션 추가)
- [ ] 섹션 간 구분이 padding만으로 되면 → FIX (배경색 전환, 시각적 구분자 추가)

**타이포그래피:**
- [ ] 모든 텍스트가 중앙 정렬이면 → FIX (좌측 정렬 기본, 의도적 중앙만)
- [ ] 제목과 본문의 크기 대비가 약하면 → FIX (제목을 더 크게)
- [ ] letter-spacing, line-height가 기본값 그대로면 → FIX (디자인 시스템 값 적용)
- [ ] Display 폰트 없이 본문 폰트만 사용하면 → FIX (폰트 페어링 적용)

**컬러:**
- [ ] 순수 #000/#fff만 사용하면 → FIX (미세한 색조 추가)
- [ ] 파랑→보라 그라디언트 사용 시 → CRITICAL (브랜드 컬러 기반으로 변경)
- [ ] 컬러 사용이 3가지 미만이면 → FIX (시맨틱 컬러 활용)
- [ ] 대비가 너무 낮아 가독성이 떨어지면 → CRITICAL

**애니메이션:**
- [ ] 애니메이션이 전혀 없으면 → CRITICAL (최소 스크롤 reveal 필요)
- [ ] 모든 요소가 동일한 fade-up만 사용하면 → FIX (다양한 패턴 혼합)
- [ ] 이징이 linear 또는 ease-in-out이면 → FIX (ease-out-expo 등 사용)
- [ ] 호버 이펙트가 opacity만이면 → FIX (scale, shadow, color 등 복합)

**디테일:**
- [ ] 호버 상태가 정의되지 않은 인터랙티브 요소가 있으면 → FIX
- [ ] 스크롤바가 기본 스타일이면 → 선택 FIX (커스텀 또는 숨김)
- [ ] 로딩/빈 상태 처리가 없으면 → FIX (스켈레톤 또는 플레이스홀더)
- [ ] 이미지가 alt 텍스트 없으면 → CRITICAL (접근성)

### Step 4: 코드 품질 검증

| 항목 | 기준 |
|------|------|
| TypeScript | 에러 0개, `any` 사용 최소화 |
| 서버/클라이언트 분리 | "use client"가 필요한 곳에만 있는지 |
| 컴포넌트 구조 | 한 파일이 200줄 초과하면 분리 검토 |
| Tailwind 사용 | 인라인 style 대신 Tailwind 클래스 사용 |
| 이미지 최적화 | next/image 사용, sizes 속성 설정 |
| 폰트 로딩 | next/font 또는 적절한 preload |
| SEO | 메타데이터 설정, 시맨틱 HTML |

### Step 5: 반응형 검증

코드에서 반응형 클래스를 확인한다:

- [ ] `sm:`, `md:`, `lg:`, `xl:` 접두사가 주요 레이아웃에 적용되었는지
- [ ] 그리드 컬럼 수가 브레이크포인트별로 변하는지
- [ ] 오버사이즈 타이포그래피가 모바일에서 적절한 크기로 줄어드는지
- [ ] 패딩/마진이 모바일에서 줄어드는지
- [ ] 네비게이션이 모바일 대응 (햄버거 메뉴 등) 되는지
- [ ] 터치 타겟이 44x44px 이상인지

### Step 6: 통합 정합성 검증

| 대상 | 왼쪽 (설계) | 오른쪽 (구현) | 확인 사항 |
|------|------------|-------------|----------|
| 섹션 구성 | 컨셉 브리프의 페이지 구성 | page.tsx의 실제 섹션 | 누락된 섹션 없는지 |
| 인터랙션 | 브리프의 인터랙션 요구 | motion/ 컴포넌트 | 요구된 인터랙션 구현됐는지 |
| 레이아웃 | 디자인 시스템의 레이아웃 컴포지션 | 섹션 컴포넌트의 그리드 | 레이아웃이 일치하는지 |

## QA 리포트 작성

`_workspace/04_qa_report.md`에 작성한다:

```markdown
# QA 리포트

## 종합 등급: [A/B/C/D/F]

| 항목 | 점수 (/10) |
|------|-----------|
| 디자인 충실도 | |
| AI 탈취 점수 | |
| 코드 품질 | |
| 반응형 | |
| 애니메이션 | |

## CRITICAL 항목 (즉시 수정)
(파일:라인 + 문제 + 수정 방법)

## FIX 항목 (수정 권장)
(파일:라인 + 문제 + 수정 방법)

## PASS 항목
(잘 된 부분 구체적으로)

## 개선 제안
(A등급으로 올리기 위한 추가 개선)
```

## 등급 기준

| 등급 | 조건 | 조치 |
|------|------|------|
| A | CRITICAL 0개, FIX 2개 이하, AI 탈취 8+점 | 완료, 사용자에게 전달 |
| B | CRITICAL 0개, FIX 5개 이하, AI 탈취 7+점 | FIX 수정 후 재검증 |
| C | CRITICAL 1~2개 또는 FIX 다수 | 수정 후 재검증 필수 |
| D | CRITICAL 3개+ 또는 AI 탈취 5점 이하 | 상당한 재작업 필요 |
| F | 빌드 실패 또는 근본적 구조 문제 | 리더에게 보고, 사용자 확인 |

## 수정 피드백 원칙

- **"이게 잘못됐다"만으로 부족하다. "이렇게 고쳐라"까지 반드시 포함한다.**
- 파일 경로와 라인 넘버를 명시한다
- 가능하면 수정 코드 스니펫을 제공한다
- 수정의 우선순위를 표시한다 (CRITICAL > FIX > 개선 제안)
